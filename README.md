# Hotspot Payment Gateway

FastAPI backend for a hotspot voucher + payment gateway. Currently implements
the auth surface (email/password + Google OAuth); payment and hotspot domains
are next.

## Stack

- Python 3.12+, [uv](https://docs.astral.sh/uv/) for dependency management
- FastAPI + Uvicorn
- SQLAlchemy 2.x (async) + asyncpg
- Alembic for migrations
- PyJWT, bcrypt, authlib (Google OAuth)
- Postgres (Neon in dev; any Postgres works)
- Pytest + httpx + aiosqlite for tests

## Prerequisites

- Python ≥ 3.12
- uv (`curl -LsSf https://astral.sh/uv/install.sh | sh`)
- A Postgres database — [Neon](https://neon.tech) is the fastest option

## Setup

```bash
# 1. Clone and install
git clone git@github.com:nadduli/hotspot-payment-gateway.git
cd hotspot-payment-gateway
uv sync

# 2. Configure environment
cp .env.example .env
# Edit .env: fill in DATABASE_URL, SECRET_KEY, optionally Google OAuth keys.
# Generate a secret with: python -c "import secrets; print(secrets.token_urlsafe(32))"

# 3. Run migrations against the direct (unpooled) Postgres endpoint
uv run alembic upgrade head
```

## Run

```bash
uv run uvicorn src.main:app --reload --port 8000
```

- API: <http://localhost:8000/api/v1/>
- Health check: <http://localhost:8000/api/v1/health>
- Interactive docs: <http://localhost:8000/docs>

## Testing & quality

```bash
uv run pytest          # full suite (currently 18 tests, in-memory SQLite)
./lint.sh              # ruff check --fix + ruff format
```

## Migrations

```bash
# Inspect / generate / apply
uv run alembic current
uv run alembic revision --autogenerate -m "short descriptive message"
uv run alembic upgrade head
uv run alembic downgrade -1
```

Migration filenames use the `YYYY-MM-DD_<slug>.py` pattern (set via
`file_template` in `alembic.ini`). The `versions/` directory is excluded
from ruff so autogenerated files aren't reformatted.


## Auth flow at a glance

```
POST /api/v1/auth/signup       -> 201 + access_token + Set-Cookie: refresh_token
POST /api/v1/auth/login        -> 200 + access_token + Set-Cookie: refresh_token
POST /api/v1/auth/refresh      -> 200 + access_token (cookie rotated)
POST /api/v1/auth/logout       -> 204 (refresh revoked, cookie cleared)
GET  /api/v1/auth/me           -> 200 + UserResponse  (Authorization: Bearer)

GET  /api/v1/auth/google/login    -> 302 to Google's consent screen
GET  /api/v1/auth/google/callback -> 302 to FRONTEND_OAUTH_CALLBACK_URI
                                     (refresh cookie set; frontend then hits /refresh)
```

Refresh tokens are HttpOnly cookies scoped to `/api/v1/auth`. Access tokens
are short-lived JWTs returned in the response body; clients pass them as
`Authorization: Bearer <token>` on subsequent requests.

## Environment variables

See [`.env.example`](./.env.example) for the full list with notes. Required
to boot:

- `DATABASE_URL` — `postgresql+asyncpg://...` (unpooled endpoint)
- `SECRET_KEY` — 32+ byte URL-safe random string

Optional but commonly set:

- `GOOGLE_CLIENT_ID` / `GOOGLE_CLIENT_SECRET` — enables `/google/*`
- `REFRESH_COOKIE_SECURE` — set to `false` for local http
- `CORS_ORIGINS` — comma-separated frontend origins
- `FRONTEND_OAUTH_CALLBACK_URI` — where the OAuth callback redirects to

## Architectural conventions

- **Package by feature.** Each feature gets its own subpackage with `models/`,
  `schemas/`, `service.py`, `router.py`, `dependencies.py`, `exceptions.py`.
  See `src/auth/` for the template.
- **SQL-first.** Joins, filters, aggregates, and nested response shaping
  belong in SQL, not Python loops. Pydantic validates the boundary.
- **Async all the way.** Engine, sessions, endpoints, tests.
- **Migrations are immutable.** Once committed, never edit a revision file;
  add a new revision instead.
- **No secrets in code or git.** `.env` is gitignored; commit `.env.example` only.
